# Contributing to QWENTUS

First off, thank you for your interest in contributing to QWENTUS! 🎉

We welcome meaningful contributions that improve the project. However, this repository maintains a high quality standard. Please read these guidelines carefully before submitting a Pull Request.

---

# Before You Start

- ⭐ Star the repository.
- 🍴 Fork the repository.
- 📖 Read the README thoroughly.
- 🔍 Check existing Issues and Pull Requests to avoid duplicate work.

---

# What You Can Contribute

We accept contributions such as:

- Bug fixes
- Performance improvements
- Documentation improvements
- New features approved through an Issue
- UI/UX enhancements
- Test coverage improvements
- Code refactoring without changing functionality

---

# Contribution Workflow

1. Fork the repository.
2. Create a new branch.

```bash
git checkout -b feature/your-feature-name
```

3. Make your changes.
4. Test your changes thoroughly.
5. Commit using meaningful commit messages.

Example:

```text
feat(parser): improve PDF extraction speed

fix(ui): resolve loading spinner issue

docs: update installation guide
```

6. Push your branch.

```bash
git push origin feature/your-feature-name
```

7. Open a Pull Request.

---

# Pull Request Requirements

Every Pull Request **must**:

- Be linked to an existing Issue.
- Solve only ONE issue.
- Pass all checks.
- Be free of merge conflicts.
- Include a clear description.
- Follow the existing project structure.
- Maintain code readability.

---

# Code Style

- Write clean, readable code.
- Use descriptive variable names.
- Keep functions small.
- Remove unused imports.
- Remove debug prints.
- Add comments only when necessary.
- Follow the formatting already used in the project.

---

# Documentation

If your contribution changes functionality, you **must** update:

- README (if applicable)
- Inline documentation
- Relevant examples

---

# What Will Be Rejected

The following Pull Requests will be closed immediately:

- ❌ Spam PRs
- ❌ AI-generated code submitted without understanding
- ❌ Formatting-only changes
- ❌ Whitespace-only changes
- ❌ README typo fixes with no real contribution
- ❌ Duplicate PRs
- ❌ Unrelated feature additions
- ❌ Massive PRs combining multiple issues
- ❌ Generated code with no explanation
- ❌ Broken or untested code

---

# Hacktoberfest Guidelines

To ensure a fair Hacktoberfest experience:

- One meaningful contribution per Pull Request.
- Low-quality PRs will receive the **invalid** or **spam** label.
- Repeated low-effort submissions may result in being blocked from contributing.

Remember:

> Quality over quantity.

---

# Code Review

Maintainers may request:

- Code improvements
- Refactoring
- Additional tests
- Documentation updates

Please respond to review comments respectfully.

---

# Branch Naming

Use one of the following:

```
feature/...
fix/...
docs/...
refactor/...
test/...
```

Examples:

```
feature/resume-ranking
fix/pdf-parser
docs/update-installation
```

---

# Commit Message Convention

Use conventional commits.

Examples:

```
feat:
fix:
docs:
refactor:
test:
style:
perf:
chore:
```

---

# Need Help?

If you're unsure where to start:

- Look for Issues labeled **good first issue**
- Ask questions in the Issue before beginning work

---

# Code of Conduct

By contributing, you agree to abide by this repository's Code of Conduct.

---

Thank you for helping make **QWENTUS** better! 🚀
